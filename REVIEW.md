# Project Review — Waymate

> Generated with the `university-project-review` skill (Fasaloft/skills). Evidence-based assessment authored for rapid project comprehension and actionable student feedback. Date: 2026-06-22.

## Project Orientation

**Waymate** is a carpooling web application built as a **Bun workspace monorepo** orchestrated by **Turborepo** (`turbo.json`, `package.json` workspaces: `apps/*`, `packages/*`, `e2e`). Package manager pinned to `bun@1.3.10`.

| Layer | Stack |
| --- | --- |
| Backend (`apps/api`) | Elysia on the Bun runtime, port 3000; **Drizzle ORM** + `postgres`; **better-auth**; **pino** logging; **Zod v4**; OpenAPI via `@elysiajs/openapi`; Vitest |
| Frontend (`apps/web`) | React 19 + Vite 7 + Tailwind CSS 4; **TanStack Router** (file-based) + **TanStack Query**; **Orval**-generated API client; **react-hook-form** + Zod; react-i18next (en/cs/sk); Vitest + Testing Library |
| Packages | `packages/db`, `packages/shared` (Zod schemas shared between API and web) |
| E2E | Playwright suite under `e2e/` (not wired into CI) |

**API module pattern** is consistent and layered: every domain module (`auth`, `users`, `cars`, `rides`, `bookings`, `reviews`, `reports`, `blocks`, `chat`, `statistics`, `health`) splits into `*.routes.ts` / `*.service.ts` / `*.repository.ts` / `*.types.ts` / `*.errors.ts`. Larger domains (`rides`, `bookings`, `reports`, `reviews`) are further decomposed into sub-folders (`creation/`, `lifecycle/`, `search/`, `listing/`, `eta/`, `admin/`).

**Database:** 14 committed migrations under `apps/api/drizzle/`, full schema in `apps/api/src/db/schema/` with status-history tables, soft deletes, and partial unique indexes. ERD and use-case docs live under `documentation/`.

This section is descriptive only; judgments follow below.

## Detailed Review

### 1. Component Library — ✅ Good

The frontend builds on the external **`@waymate/ui`** library (`apps/web/package.json` → `"@waymate/ui": "^0.1.59"`) plus **Radix UI** primitives (`@radix-ui/react-dropdown-menu`, `@radix-ui/react-select`).

Raw HTML element bypass is essentially absent: across `apps/web/src` there are **0** `<button>` and **1** `<input>` usages in `.tsx` files. Interaction elements come from the design system rather than bare HTML.

**Recommendations:** none material. 1. Optionally audit the single raw `<input>` to confirm it should not use the `@waymate/ui` field component.

### 2. Styling — ✅ Good

Tailwind CSS 4 is the styling system. Anti-patterns are minimal:

- Inline `style={{…}}`: **1** occurrence total across `.tsx` files.
- `!important`: **2** occurrences.
- Tailwind arbitrary values: only one spotted (`[10px]` in `apps/web/src/components/navigation/AdminNavbar.tsx:125`).

**Recommendations:** 1. Replace the `[10px]` arbitrary value with a theme spacing token if one fits. 2. Confirm the two `!important` rules are unavoidable overrides of third-party styles.

### 3. Loading Data — ✅ Good

Data access is centralized through **TanStack Query** hooks generated by **Orval** into `apps/web/src/api-client/`, consumed via the custom fetcher `apps/web/src/lib/api-fetcher.ts` (injects `credentials: "include"`, throws `ApiError`). A grep for direct `fetch(`/`axios` calls inside `.tsx` components returns **none** (excluding `refetch`/`prefetch`). Mutations are wrapped in thin hooks (e.g. `apps/web/src/features/passenger/hooks/useCreateBooking.ts`, `apps/web/src/hooks/shared/useSubmitReview.ts`) that own cache invalidation.

**Recommendations:** none.

### 4. Environment Variables — ✅ Good

`.gitignore` ignores `.env` / `.env.*` while explicitly allowing `!.env.example` and `!.env.test.example`, and additionally ignores `bunfig.toml` (noted "contains auth tokens") and `.claude/settings.local.json`. A thorough `apps/api/.env.example` documents every variable.

Configuration is **validated at boot** through Zod in `apps/api/src/config/env.ts`: required `DATABASE_URL`, `BETTER_AUTH_URL`, `WEB_ORIGIN`, `RESEND_API_KEY`; a custom `OriginSchema` rejects origins carrying a path/query/fragment so CORS misconfiguration fails loud. No hardcoded connection strings, ports, or keys appear in source — the only string URL literals are documented defaults (`OSRM_BASE_URL` default `https://router.project-osrm.org` at `env.ts:91`).

**Recommendations:** none.

### 5. REST API Design — ✅ Good

Routes are resource-oriented plural collections (`/rides`, `/bookings`, `/cars`, …). The codebase enforces a deliberate convention (documented in `CLAUDE.md`, visible in routes): **state-machine transitions use `PATCH /<collection>/:id/<action>`** (e.g. `PATCH /rides/:id/{cancel,end,complete}`, `PATCH /bookings/:id/{cancel,confirm,reject}`), creation is `POST /<collection>` → `201`, and the single non-collection `POST /rides/estimate-eta` is a clearly read-only complex-body query.

Request/response validation uses **Zod v4** schemas defined in `packages/shared` and registered as Elysia models — see `apps/api/src/modules/rides/ride.routes.ts`, which declares typed `response: { 200, 429, 500 }` shapes per route. Status semantics are explicit per domain (see category 11).

**Recommendations:** none.

### 6. Database — ✅ Good

Schema is well-normalized Drizzle (`apps/api/src/db/schema/`): **19** files use `references(...)` for foreign keys; UUID primary keys via `gen_random_uuid()` throughout. **14** committed migrations (`drizzle/0000_*.sql` … `0013`) provide a versioned, audited history, and CI has a `migration-drift` job to keep SQL in sync with the schema.

Images are stored as a **URL/text reference** (`user.ts:22 image: text("image")`), not as a blob — correct. Schema-level integrity is strong: CHECK constraints (`reviews_rating_chk` rating BETWEEN 1 AND 5, author ≠ subject, non-negative prices), partial unique indexes scoped to `WHERE deleted_at IS NULL`, and three status-history tables for audit trails. Raw `sql\`\`` usage is confined to schema constraints and the seed's TRUNCATE — all parameterized via tagged templates, no string concatenation.

**Recommendations:** 1. `apps/api/src/db/seed.ts` is 1230 lines — consider splitting fixture groups into modules for readability (dev-only, low priority).

### 7. Backend Design Patterns — ✅ Good

Layer separation is strict and consistently applied. Repositories are pure Drizzle queries taking an `Executor` (`db | tx`) first; services are the only layer importing `db` and the only layer calling `db.transaction(...)`; routes only map domain errors to HTTP. This is visible across every module and reinforced in `CLAUDE.md`. Complex multi-table flows (ride creation, booking lifecycle) are wrapped in transactions inside the service layer (`apps/api/src/modules/rides/creation/ride-creation.service.ts`, `bookings/lifecycle/`).

Business logic does not leak into route handlers — `ride.routes.ts` handlers delegate straight to `RideService`.

**Recommendations:** none.

### 8. Auth — ✅ Good

Authentication uses **better-auth** with the Drizzle adapter (`apps/api/src/modules/auth/auth.ts`), supporting email/password and conditional Google OAuth (enabled only when both client id/secret are set). A grep for `jsonwebtoken` / `bcrypt` / `jose` / hand-rolled HMAC returns **nothing** — no custom crypto, which is the desired outcome.

Authorization goes **beyond authentication**: three composable Elysia macros (`isAuthenticated`, `isFullyOnboarded`, `requireAdmin` in `auth.middleware.ts`) enforce session, onboarding, and role. Role is non-user-settable (`input: false`), and admin endpoints filter out admin rows. Ownership checks exist at the domain layer (e.g. `RideNotFoundOrNotOwner`).

**Recommendations:** none.

### 9. Testing — ⚠️ Concerns

The backend is well covered: **26** `*.test.ts` files in `apps/api` spanning routes, services, repositories, middleware, rate-limiter, and time utilities, run against a throwaway Postgres in CI. **No** `.skip` / `.todo` / `.only` were found — coverage is real, not stubbed.

The concern is **balance**: the React frontend has only **5** test files (`apps/web/src/**`) for a large app (~60 route/component files), and the **Playwright e2e suite is explicitly excluded from CI** (`.gitlab-ci.yml` runs unit/integration only). For a full-stack project this leaves UI behavior and end-to-end flows under-verified relative to the API.

**Recommendations:** 1. Add component/hook tests for the highest-value flows (booking, ride offer form, chat). 2. Wire at least a smoke subset of the Playwright e2e suite into CI so it doesn't rot. 3. Track frontend coverage with `@vitest/coverage-v8` (already a dependency on the API side).

### 10. Logging & Monitoring — ✅ Good

Structured logging via **pino** (`apps/api/src/shared/logger.ts`): JSON in production, `pino-pretty` in development, level via `LOG_LEVEL`. **Sensitive-data redaction is configured** for Authorization/Cookie/Set-Cookie headers and `*.password`/`*.token` fields. Per-request lifecycle logging mints a `requestId`, echoes `x-request-id`, and emits one structured `request` line with `{ method, path, status, durationMs }`; 500-class errors are logged with stack + requestId.

`console.*` usage in production paths is **1** occurrence (`src/openapi/post-process.ts:183` `console.warn`); the rest are confined to CLI scripts (`seed.ts`, `reset.ts`) and the pre-logger env-validation failure path, which is the documented and correct exception.

**Recommendations:** 1. Consider routing the single `console.warn` in `post-process.ts` through the pino logger for consistency (minor; it runs at build/spec-dump time).

### 11. Error Handling — ✅ Good

Errors are modeled as typed domain errors per module (`ride.errors.ts`, `auth.errors.ts`, etc.) extending a shared `DomainError`, with an explicit `…ErrorToHttpStatus` mapper — e.g. `rideErrorToHttpStatus` maps `RideNotFound` → 404, `CarNotAvailableForDriver` → 403, validation/state errors → 400, with an `assertNever` exhaustiveness guard. Macros and services **throw**; only `.onError` translates to HTTP. No empty `catch {}` blocks were found in the API.

Frontend renders error states: a `RouteErrorBoundary` component (with its own test) plus **42** files referencing `isError` / `errorComponent` / the boundary. The two `try/catch` swallows in `ride.routes.ts` (`resolveViewerId`) are intentional and commented — anonymous viewers must not throw.

**Recommendations:** none.

### 12. Security — ✅ Good

- **CORS is an explicit allowlist, not a wildcard**: `allowedOrigins = Array.from(new Set([env.WEB_ORIGIN, ...env.CORS_ORIGINS]))` (`apps/api/src/index.ts:52`), with `credentials: true` and a fixed method/header list. Origins are Zod-validated as bare origins.
- **No hardcoded secrets** in source; all secrets come through validated env.
- **Parameterized queries** exclusively via Drizzle ORM / `sql\`\`` tagged templates — no string interpolation into SQL.
- **Request hardening**: body-size limit (413 `PAYLOAD_TOO_LARGE`) and an in-memory fixed-window rate limiter (global 60/min + stricter per-route caps, 429 + `Retry-After`), with client IP read from `X-Forwarded-For` at the `TRUSTED_PROXY_COUNT` position so it can't be spoofed.
- `.gitignore` protects `.env*`, `bunfig.toml` (auth tokens), and local Claude settings.

**Recommendations:** 1. As `CLAUDE.md` already notes, the rate limiter is per-process — back it with Redis before running multiple API replicas in production. 2. Consider promoting body-size/rate-limit constants to env vars so ops can tune without redeploy.

### 13. Forms — ✅ Good

Forms use **react-hook-form** (`useForm` in **19** files) with **Zod** validation via `@hookform/resolvers/zod` (`zodResolver` in **12** files). Example: `apps/web/src/routes/driver/offer/index.tsx` wires `useForm` + `zodResolver(offerRideSchema)` + `FormProvider`, with server errors fed back through `setError`. A dedicated `FieldError` component handles message rendering. No evidence of the excessive-`useState`-per-field anti-pattern in form components.

**Recommendations:** none.

### 14. Frontend Structure — ⚠️ Concerns

The frontend uses **TanStack Router file-based routing** with a clean `routes/` tree and co-located `-components/`, `-hooks/`, `-lib/`, `-schema.ts` folders — a good cohesion pattern. Data fetching is extracted into hooks (`-hooks/useOfferRideSubmit.ts`, `useDriverDashboardData.ts`, etc.) rather than inlined in route bodies. Most route components stay reasonably small.

Two concerns:

1. **A handful of files exceed the 300-line guideline**, signalling decomposition opportunities: `src/lib/geocoding/photon.ts` (354), `src/routes/driver/offer/-hooks/useOfferRideSubmit.ts` (350), `src/routes/admin/rides/-components/RideDetailModal.tsx` (332), `src/features/chat/components/ChatPanel.tsx` (330), `src/routes/passenger/rides/index.tsx` (320), `src/routes/driver/offer/-components/CarSection.tsx` (320), `src/routes/admin/reports/-components/ReportDetailModal.tsx` (308), `src/components/navigation/AdminNavbar.tsx` (307). None are egregious (>400), but the detail modals and `ChatPanel` are candidates for sub-component extraction.
2. **Documentation drift:** `CLAUDE.md` describes routing as "code-based routing in `apps/web/src/router.tsx`", but the project has migrated to **file-based** routing (`routes/` + generated `routeTree.gen.ts`). The doc should be updated so it matches reality.

**Recommendations:** 1. Decompose the 300+ line detail modals and `ChatPanel` into smaller presentational sub-components. 2. Update the routing section of `CLAUDE.md` to reflect the file-based TanStack Router setup. 3. Confirm `photon.ts` (geocoding lib helper) belongs in `lib/` and split its concerns (request building vs. response mapping) if it keeps growing.

## Closing Note for the Lecturer

Waymate demonstrates production-grade engineering discipline rare in a student project: strict service/repository/route layering, boot-time env validation, parameterized ORM queries, redacted structured logging, an explicit CORS allowlist, request hardening (rate limiting + body caps), and an audited migration history with status-history tables. The substantive gaps are **frontend test depth** and **e2e in CI** (category 9), plus minor **file-size and doc-drift** cleanups (category 14). Both are improvement areas with clear remediation paths, not correctness or security risks.
